    Configuration File¶

    Example configuration file:

    protection:
  names:
    enabled: true
  control-flow:
    enabled: true
    logic:
      enabled: true
      allowlists:
        - level: medium
          allowlist:
            - "EncryptFunc"
  code-integrity:
    enabled: true
    functions:
      enabled: true
  environment-integrity:
    enabled: true
    debugger:
      enabled: true
    root:
      enabled: true
    jailbreak:
      enabled: true

    Use clean indentation!

    The configuration uses the YAML format. Similar to, for example the Python language, indentation has meaning.

    Features¶

    Metadata Obfuscation¶

    Flutter apps are written in Dart. When compiling Flutter apps in release mode, the ahead-of-time compiler is used to generate a native app. This app contains a Dart snapshot with metadata. This metadata contains information about objects, functions, data (eg. strings), etc.

    Flutter is able to obfuscate the metadata and rename all function names. This is already a good first step but is not enough. For a reverse engineer this metadata is still valuable because:

    • It contains all of the strings that are used in the app and,
    • The metadata can still be used to identify function boundaries and decompile them more easily.

    DexGuard or iXGuard protects your app by encrypting this valuable metadata and your strings.

    Compatibility with Flutter built-in obfuscation¶

    Flutter provides a simple built-in obfuscation capability with the --obfuscate switch.

    We recommend to switch the built-in obfuscation off when protecting your Flutter product with Guardsquare. While having using both at the same time will technically work, it will not provide extra reverse engineering resilience, but would make it difficult to read the stack traces such as crash logs. This is due to the fact that Guardsquare does not support stack trace symbolication for Flutter built-in obfuscation.

    Configuration¶

    enabled¶

    To start using metadata obfuscation, enable it in the configuration under the names key:

    Default: false

    names:
  enabled: true

    Verification¶

    You can verify the result by running strings on the libapp.so or App.framework/App file in your unzipped .apk/.ipa before and after obfuscation to see the difference.

    # Android
$ unzip -d unpacked_apk my_flutter_app.apk
$ strings unpacked_apk/lib/arm64-v8a/libapp.so

# iOS
$ unzip -d unpacked_ipa my_flutter_app.ipa
$ strings unpacked_ipa/Payload/Runner.app/Frameworks/App.framework/App

    Control-flow Obfuscation¶

    Just like any other app, written in any language, Flutter apps written in Dart code contains application logic and algorithms that are sometimes themselves important intellectual property. In other cases reversing the logic is a step towards targeted tampering with the application code.

    The different aspects of an application's control flow can be obfuscated.

    Configuration¶

    enabled¶

    Enable or disable techniques for obfuscation of function logic under the control-flow.logic key. Note that this feature requires a allowlist. Please refer to the Filters section for more information on how to specify it.

    Default: true

      control-flow:
    enabled: true
    logic:
      enabled: true
    allowlists¶

    The allowlist allows you to specify which functions or methods are considered for control flow obfuscation. You can again use regular expressions here to simplify configuration. level indicates the strength of the applied obfuscation techniques. This currently makes no difference when applied to Dart code.

    Default: none

    control-flow:
  enabled: true
  logic:
    enabled: true
    allowlists:
      - level: medium
        allowlist:
          - "RSA.*"
          - "DecryptFunction"

    Code Integrity¶

    A common goal for attackers is changing the behaviour of your application, this could be the end goal or the means to an end. For example, removing a license check could be a final goal whereas printing decrypted network traffic could be a step towards analysing a remote API.

    These kinds of changes to application behaviour usually require adapting the code, which can be done through static binary patching or dynamically during execution. The latter is often done with the help of tools like Cydia Substrate or advanced Dynamic Binary Instrumentation toolkits like Frida or HookZz.

    These tools can also be applied to change the behavior of Dart code. DexGuard and iXGuard offer protection against inline hooking of Dart functions.

    Configuration¶

    enabled¶

    Enable or disable protection against inline hooking under the code-integrity.functions key. This protects against hooking techniques that will change the start of the function body (eg. Frida).

    Default: true

    code-integrity:
  enabled: true
  functions:
    enabled: true
    allowlist¶

    Inline hooking protection automatically selects functions for protection. To make sure that a function is always protected, it can be specified in an allowlist.

    Default: none

    code-integrity:
  enabled: true
  functions:
    enabled: true
    allowlist:
      - "someMethod"
    denylist¶

    The denylist allows you to specify functions where the checks must not be injected and which functions will not be protected. Please refer to the Filters section for more information on how to specify it.

    Default: none

    code-integrity:
  enabled: true
  functions:
    enabled: true
    denylist:
      - "dontProtectMe"
    aggressiveness¶

    Default: medium

    code-integrity:
  enabled: true
  functions:
    enabled: true
     aggressiveness: medium

    Use aggressiveness parameter to determine the amount of the checking code injected. Aggressiveness can have one of the 5 available levels: min, low, medium, high, max. Be aware that picking a higher level here can have a significant impact on application size and performance.

    callback¶
    code-integrity:
  enabled: true
  functions:
    enabled: true
    callback: "myCallback"
    continue-on-callback: false

    A callback can be defined to configure a custom reaction to integrity violations. Refer to Callbacks for more information.

    Environment Integrity¶

    Runtime Application Self Protection (RASP) remains an important cornerstone to protect your applications from reverse engineering attempts. This is again regardless of the language that your app is written in. Therefore, environment checks can be injected in to your Dart code to optimally help protect your Flutter app. By adding this on top of the existing Guardsquare RASP checks, we add an additional layer of protection to thwart reverse engineers.

    Configuration¶

    Root, jailbreak and debugger detection are separate options to enable and configure. To inject both you need to set both options:

    Note

    Root detection only applies to Android apps and is ignored for iOS apps. The same is true for jailbreak detection when processing Android apps.

    Danger

    The root detection checks injected in Flutter code cannot detect all types of rooting setups. To achieve a more complete root detection coverage, also enable root detection in DexGuard.

    environment-integrity:
  enabled: true
  debugger:
    enabled: true
  jailbreak:
    enabled: true
  root:
    enabled: true
    denylist¶

    The denylist allows you to specify functions where the RASP checks must not be injected. Please refer to the Filters section for more information on how to specify it.

    Default: none

    environment-integrity:
  debugger:
    enabled: true
    denylist:
      - "prepareService.*"
    checkpoints¶

    Checkpoint functions can be defined explicitly and it is guaranteed that the specific check will be performed before the function is executed.

    Default: none

    environment-integrity:
  debugger:
    enabled: true
    checkpoints:
      - "encrypt"
      - "decrypt"
    spray¶

    To insert checks randomly in the code, use the spray policy. The detection logic will be placed in all available code that is not denylisted.

    Spraying runtime checks makes it significantly more difficult to patch them out manually or automatically due to the following:

    1. Diversity. Every check will be obfuscated differently making it difficult to locate them automatically.

    2. Amount. A large number of runtime checks will be distributed in the code automatically making it difficult to patch all of them manually.

    3. Randomness. Given a large injection base, every build will place the checks differently, making it impossible for an attacker to automate based on older builds.

    Use aggressiveness parameter to determine the amount of the checking code injected. Aggressiveness can have one of the 5 available levels: min, low, medium, high, max. Be aware that picking a higher level here can have a significant impact on application size and performance.

    Default: true medium

    Example configuration for debugger protection:

    environment-integrity:
  enabled: true
  debugger:
    enabled: true
    spray: true
    aggressiveness: medium
    callback¶
    environment-integrity:
  enabled: true
  debugger:
    enabled: true
    callback: "myCallback"
    continue-on-callback: false

    A callback can be defined to configure a custom reaction to integrity violations. Refer to Callbacks for more information.
