    Migration to automatic RASP injection¶

    If this migration guide does not cover one of your use cases, please reach out to us via support.

    We are deprecating the runtime library in favor of automatic RASP injection. Automatic RASP injection is more secure and relieves you from the burden of implementing calls to the runtime library and handling the results. With automatic RASP injection, you should be able to replace your existing implementation of the runtime calls as well as configure DexGuard to inject RASP checks at other security sensitive locations in the app automatically.

    Moving forward, there are two automatic RASP injection options to replace the DexGuard runtime library:

    1. (Recommended) Using entrypoint and Profile-guided obfuscation (PGO) based spray injection. You can profile your app for the PGO based spray injection using the guided configuration workflow.
    2. In case option 1 is not the most viable solution for your app, you can use checkpoint injections.

    This page provides guidance on how to migrate from using the DexGuard runtime library to automatic RASP injection.

    Remove dependency on the DexGuard runtime AAR¶

    Automatic RASP does not require apps or SDKs to depend on the DexGuard runtime AAR. Therefore, the following can be removed from your build files.

    build.gradle(.kts)
    dependencies {
    // Example: include the DexGuard runtime library.
    implementation(libs.dexguard.runtime)
}
    gradle/libs.versions.toml
    [versions]
dexguard = "9.10.5"

[libraries]
dexguard-runtime = { group = "com.guardsquare", name = "dexguard-runtime", version.ref = "dexguard" }

    build.gradle(.kts)
    dependencies {
    // Example: include the DexGuard runtime library.
    implementation('com.guardsquare:dexguard-runtime:9.10.5')
}

    Replace runtime calls with checkpoints¶

    Manual calls to the runtime library can be replaced by RASP checkpoints. The following is a generic form of the RASP checkpoint configuration option.

    -raspchecks <detector> checkpoint class <class-specification>
    Here is an example of a runtime library call:

    class HelloWorldActivity : Activity() {
    companion object {
        /**
        * The method where a call to the runtime is made.
        */
        fun myMethod() {
            Log.i("HelloWorldActivity", "This method calls the runtime.")
            val OK = 1
            val returnValue = HookDetector.isApplicationHooked(OK)
            if (OK != returnValue) {
                // Reaction to a triggered check.
            }
        }
    }
}

    public class HelloWorldActivity extends Activity {
    /**
    * The method where a call to the runtime is made.
    */
    public void myMethod() {
        Log.i("HelloWorldActivity", "This method calls the runtime.");
        int OK = 1;
        int returnValue = HookDetector.isApplicationHooked(OK);
        if (OK != returnValue) {
            // Reaction to a triggered check.
        }
    }
}

    This can be replaced with the following configuration option:

    -raspchecks hook checkpoint class HelloWorldActivity {
    public void myMethod();
}

    By default, DexGuard may elect to inject the check into a method that always precedes the target method. To disable this behavior and ensure the check is always injected exactly in the specified method, you can use the -disableraspdominator configuration option.

    Use this only as a transition measure

    This approach closely resembles the use of the runtime library, but it does not give you all the security benefits of a complete RASP injection strategy. Because of this, we advise to use this only as a temporary transition towards fully utilizing auto injected RASP. A good next step would be to enable entrypoint injection. You can then look into identifying other sensitive methods and configuring them as checkpoints as well, or configure spray injection. When configuring spray injection, we suggest using Profile-Guided Obfuscation, which will minimize the performance impact of the injected RASP checks.

    Passing a flag to a RASP check¶

    If you are passing a flag in any of the runtime library calls, you can replace it with the following configuration option:

    -raspflags <detector> <flag-number>

    Here <detector> is the RASP check, e.g., hook, root, etc., and <flag-number> is the integer value of the flag from the runtime library. Please reach out to us via support if you are using a flag and you are not sure about its integer value. If you pass multiple flags to the same RASP check, then you need to provide the option only once with the values bitwise ORed together.

    Passing a certificate hash value to the CertificateChecker¶

    In certain cases such as Google Play Store app signing, building app bundles or app resigning, a certificate hash needs to be provided to the CertificateChecker. In automatic RASP, the same can be achieved using the -raspcertificatehash configuration option.

    Replace manual reaction with RASP callbacks¶

    Any reaction to a positive runtime check can be replaced with configuring a RASP callback and implementing the reaction, e.g., sending the event to Crashlytics, triggering a UI reaction, etc., inside the RASP callback.

    The following code snippet is a template of how a reaction to a triggered check is implemented.

    val OK = 1
val returnValue = HookDetector.isApplicationHooked(OK)
if (OK != returnValue) {
    // Reaction to a triggered check.
}

    int OK = 1;
int returnValue = HookDetector.isApplicationHooked(OK);
if (OK != returnValue) {
    // Reaction to a triggered check.
}

    This can be replaced by implementing and configuring a RASP callback.

    class HelloWorldActivity : Activity() {
    companion object {
        /**
        * This is the callback method which will be called when a RASP check is triggered.
        *
        * @param debugInfo A long value containing encoded threat information.
        */
        @JvmStatic
        fun myRaspCallback(debugInfo: Long) {
            // Reaction to a triggered check.
        }
    }
}

    public class HelloWorldActivity extends Activity {
    /**
    * This is the callback method which will be called when a RASP check is triggered.
    *
    * @param debugInfo A long value containing encoded threat information.
    */
    public static void myRaspCallback(long debugInfo) {
        // Reaction to a triggered check.
    }
}

    The callback method needs to be configured in your DexGuard configuration.

    -raspcallback hook class HelloWorldActivity {
    public static void myRaspCallback(long);
}

    Debug info about triggered RASP checks¶

    In the runtime library checks, the OK value passed to a runtime API call and its return value helps in understanding the detection and investigating potential false positives. With automatic RASP, this can be achieved in two ways:

    • With ThreatCast: The threat event is reported to ThreatCast and displayed on the dashboard with proper explanation. Therefore, no additional information is required from the customer side.

    • Without ThreatCast: If ThreatCast isn't configured, a callback must be configured to retrieve and share the debugInfo value. The debugInfo value is passed to the callback method.

    Crash on detection¶

    It is highly discouraged to implement a custom crashing reaction to a positive RASP check. You can utilize DexGuard-initiated app terminations instead, which are inherently more secure and effective.
