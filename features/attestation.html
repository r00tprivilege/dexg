    App attestation¶

    Introduction¶

    App attestation allows you to ensure that requests to your APIs are originating from legitimate/trusted devices. It is an extension of ThreatCast and makes the collected insights about users and devices actionable.

    The app attestation flow is as follows:

    1. While your application is running, data is gathered on the device.
    2. The collected data is sent to the ThreatCast server.
    3. The data is analyzed on the ThreatCast server, and a verdict on the authenticity and security of the application and its environment is communicated to the application by means of an encrypted token.
    4. Your application attaches the token to any requests sent to your API.
    5. Your application server decrypts the token and decides how to respond to the API request.

    The remainder of this page contains instructions on how to set up and use app attestation in your application.

    Quickstart¶

    Prerequisites¶

    App attestation requires that ThreatCast is enabled for your application, as well as a new v2 ThreatCast API key. In order to encrypt tokens in a way that allows only your server to decrypt them, you must also supply a public encryption key. The following sections will guide you through these steps.

    Obtaining a V2 API key¶

    You can manage your ThreatCast API keys in your ThreatCast app settings (App Settings -> API Keys). Here, either select an existing v2 key, or create a new v2 key.

    Configuring your encryption key¶

    Tokens issued by the ThreatCast server will be encrypted using a public key that you provide. Only you will be able to decrypt the tokens on your server, using the matching private key.

    To configure your public key, first generate a 4096 bit RSA keypair. The exact command will depend on your tooling and platform. We provide an example using the openssl command below:

    OpenSSL RSA key generation
    openssl genpkey -algorithm RSA -out private_key.pem -pkeyopt rsa_keygen_bits:4096
openssl rsa -pubout -in private_key.pem -out public_key.pem
    If you are using a different tool, please ensure that the public key you provide on ThreatCast is a 4096-bit RSA-key in the PEM encoding of its PKIX, ASN.1 DER form.

    Once you have obtained your keypair, navigate to the ThreatCast page for the app you are enabling attestation on. Go to App settings, then Keys, and click the Upload key button to bring up the key entry window. Inside of this window, paste the contents of the public_key.pem file that was previously created. Finally, click the "upload" button to complete the process.

    Configuration¶

    See the AppAttestation sample in your DexGuard distribution for an illustration of these steps.

    You can find this sample in your DexGuard distribution at samples/features/AppAttestation.

    First, add a dependency on the attestation client library in your application.

    app/build.gradle.kts
    repositories {
    maven {
        credentials {
            username = "maven"
            password = "${properties["dexguardMavenToken"]}"
        }
        url = uri("https://maven.guardsquare.com")
        // Only search for artifacts with groupId "com.guardsquare.*",
        // supported since gradle 5.1.
        content {
            includeGroupByRegex("com\\.guardsquare.*")
        }
        authentication {
            create<BasicAuthentication>("basic")
        }
    }
}

dependencies {
    implementation("com.guardsquare:dexguard-attestation-client:9.10.5")
}
    app/build.gradle
    repositories {
    maven {
        credentials {
            username = "maven"
            password = "${dexguardMavenToken}"
        }
        url "https://maven.guardsquare.com"
        // Only search for artifacts with groupId "com.guardsquare.*",
        // supported since gradle 5.1.
        content {
            includeGroupByRegex "com\\.guardsquare.*"
        }
        authentication {
            basic(BasicAuthentication)
        }
    }
}

dependencies {
    implementation("com.guardsquare:dexguard-attestation-client:9.10.5")
}
    app/build.gradle.kts
    repositories {
    flatDir { dirs("/path/to/DexGuard/lib") }
}
dependencies {
    implementation(mapOf("name" to "dexguard-attestation-client", "ext"  to "aar"))
}
    app/build.gradle
    repositories {
    flatDir { dirs "/path/to/DexGuard/lib" }
}
dependencies {
    implementation(name: 'dexguard-attestation-client', ext: 'aar')
}

    Next, enable app attestation by adding the -appattestation option to your DexGuard configuration. Make sure to also add your v2 API key for ThreatCast if you haven't done so already.

    DexGuard configuration file
    -threatcastapikey YOUR_V2_API_KEY
-appattestation

    After these steps, everything is in place to start using app attestation in your application.

    Given the above minimal configuration, DexGuard will implicitly inject all RASP checks except OS integrity, debug blocker, and certificate tamper (unless -raspcertificatehash is specified) into your application. We do this to make sure the verdict you receive from app attestation is of the highest quality possible using all of the currently available capabilities without relying on having an optimal RASP configuration. The checks are injected in such a way that they will not affect performance in a perceivable way.

    App attestation is currently not supported for libraries.

    Implementation¶

    Using app attestation involves the following steps:

    1. Fetch an attestation token from the ThreatCast server.
    2. Attach the attestation token to your API requests.
    3. Analyze the attestation token on your application server.

    See the AppAttestation sample in your DexGuard distribution for an illustration of these steps.

    Let's have a closer look at each of these steps.

    Fetching an attestation token¶

    DexGuard automatically takes care of the data collection on the device, but analysis of the data needs to be explicitly requested from the ThreatCast server.

    This is done using the getToken method provided in the attestation client. This getToken method sends all the collected data to the ThreatCast server, prompts analysis of the data, and returns the encrypted attestation token.

    What happens in case the ThreatCast server is unreachable?

    The getToken network calls have a timeout of 5 seconds. If no token is returned by the ThreatCast server within this duration, a locally-generated server offline token is returned. The server-side SDK should be used to verify on your server whether ThreatCast was indeed unreachable.

    Two APIs for the getToken method are available in the com.guardsquare.dexguard.attestation package: 

    String AppAttestation.getToken(Context context)
String AppAttestation.getToken(Context context, String requestHash)
    The second API allows you to specify a unique hash for the payload when requesting a token, such that the token is bound to the payload.

    In case you are also implementing attestation on iOS and would like the code to look as similar as possible, you can also consider using the following alternative APIs, which are identical to the above, but follow the naming convention used for native code. 

    String AppAttestation.GSGetAttestationToken(Context context)
String AppAttestation.GSGetAttestationTokenWithRequestHash(Context context, String requestHash)

    Payload binding allows your server to confirm that a token has not been replayed.

    The hash that is generated for payload binding must be unique to the network request and your server must be able to generate the same unique hash on its end by using the contents of the received network request.

    The attached token will include this hash so that your server can decode it and verify it against the hash it calculates itself. This ensures that the token can only be used to validate that specific payload and has not been reused multiple times.

    Sending the attestation token to your application server¶

    In order to always have an up-to-date token available when your app makes an API request, it is advised to add calls to getToken at all points in your app where you want to attest such a request.

    Token validity should be checked against the expiration timestamp.

    Every token that is received by your server will contain an expiry time within the decoded and decrypted attestation token. Your server should check this value on the token to ensure it is not expired.

    Don't request tokens on the main thread!

    Since getToken calls are network requests to the ThreatCast server, they must be made on a background thread to prevent blocking the main thread. If they are made in the main thread, a NetworkOnMainThreadException will be thrown. Please make sure to thoroughly test your app and API boundaries to ensure these calls are always made on background threads.

    You cannot request a token from an Application class!

    In an Application class, i.e. a class that extends android.app.Application, the context is not fully initialized to perform a token request. Thus, a call to getToken will fail and an exception will be thrown.

    The token retrieved by the getToken call now needs to be attached to your app's API requests in order to be verified on the application server. Only then can your server verify whether the app that performs the request is permitted to perform some action, such as access certain data.

    You can attach these tokens either using a simple header in the network request, or by adding the token to the payload of the API request.

    Debugging app attestation¶

    Your DexGuard distribution also includes a debug attestation client you may use to test and debug your attestation setup. You can use this by including the debug option -useattestationdebugclient in your configuration, in addition to the -appattestation option.

    DexGuard configuration file
    -useattestationdebugclient

    The debug client will output runtime logs about when requests to your server are made and received, as well as the size of the requests. After debugging, you should remove the configuration option and rebuild your application.

    Logging the token

    If you choose to log the returned token, Android runtime logs will truncate the token by default due to the maximum limit on logcat messages. You will need to ensure any debug logging produces the full token.

    Don't use the debug client in your release builds!

    The app will produce sensitive logs about the attestation requests, which should not be exposed to customers.

    Shell attestation client¶

    The dexguard-attestation-client.aar artifact does not contain the actual attestation implementations. The artifact is a shell that is replaced when processing with DexGuard (assuming -dontobfuscate isn't set).

    Warning

    Using the dexguard-attestation-client.aar artifact with an unprotected app is still possible but no attestation will be performed.

    Warnings have been set up to make sure that you are aware that you are using the shell library:

    • Logs will be printed to logcat every time you invoke a getToken call.
    • A toast will be displayed the first time it is invoked.

    You can remove the toast by adding the following meta-data to your application tag of the AndroidManifest.xml file. By setting this metadata you explicitly accept the fact that you are using a shell library.

    Example: 

    <application
        android:label="@string/app_name"
        [...]>

        <meta-data android:name="hideAttestationShellWarning" android:value="true"/>

        [...]
</application>

    Note: The metadata will automatically be removed from your release build when including one of our default release configurations.

    Security considerations¶

    SSL Pinning¶

    It is advised to use SSL pinning on the APIs your application will call on your server. See the Android documentation on this for more information.

    Accessibility malware protection¶

    When implementing app attestation for protection against malicious accessibility services, it is vital that you launch your sensitive activities only after validating the token! As soon as an activity is launched, malicious accessibility services may have access to all displayed information!

    Pseudocode example¶

    Below pseudocode demonstrates the flow we suggest to adopt.

    async launchSensitiveActivity(val sensitiveActivityIntent: Intent) {
    // 1. Get the attestation token using the attestation API
    val token = getToken()

    // 2. Forward the token to your application server.
    val malwareInstalled = validateToken(token)

    // 3. Only launch your activity if your policy was validated.
    if (!malwareInstalled) startActivity(sensitiveActivityIntent)
}

async validateToken(val token: Token) : Boolean {
    // This method forwards the token to your application server,
    // and returns whether your malware policy was violated or not. 
}

    A setup such as this guarantees that sensitive activities are not started unless your defined policy is validated.

    Keep in mind that the getToken and validateToken calls are asynchronous networking calls which cannot be performed on the main thread.
