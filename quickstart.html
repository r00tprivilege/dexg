    Manual Configuration

    Prepare your development environment¶

    DexGuard is available through Maven. If you plan to integrate DexGuard in your Gradle build process with Maven, please follow the instructions on the Maven configuration page and skip forward to the DexGuard configuration step.

    If you want to manage your DexGuard installation locally, make sure you have the right tools and components in place.

    1. Log in to the Guardsquare website.
    2. Download the DexGuard software and your license file dexguard-license.txt and install them on your local system.
    Not using Gradle?

    If you don't want to integrate DexGuard in your Gradle build, or you are not using Gradle at all, you can run DexGuard directly from the command line. Detailed instructions can be found in the standalone mode page. Please follow these instructions and skip forward to the automated configuration step.

    If you are using hybrid app development, see also the technology guidance for

    • Flutter apps
    • JavaScript hybrid apps

    Setting up the Gradle plugin¶

    The DexGuard Gradle plugin is available in the lib folder of your DexGuard distribution. It can easily be added to your project by making the changes highlighted in the code snippet below.

    settings.gradle.kts
    pluginManagement {
    repositories {
        maven { url=uri("/path/to/DexGuard/lib") }
    }
}
    gradle/libs.versions.toml
    [versions]
dexguard = "+"

[plugins]
dexguard = { id = "com.guardsquare.dexguard", version.ref = "dexguard" }

    build.gradle.kts
    buildscript {
    repositories {
        maven { url=uri("/path/to/DexGuard/lib") }
    }
    dependencies {
        classpath("com.guardsquare:dexguard-gradle-plugin:+")
    }
}

    settings.gradle
    pluginManagement {
    repositories {
        maven { url '/path/to/DexGuard-x.y.z/lib' }
    }
}
    gradle/libs.versions.toml
    [versions]
dexguard = "+"

[plugins]
dexguard = { id = "com.guardsquare.dexguard", version.ref = "dexguard" }

    build.gradle
    buildscript {
  repositories {
    maven {
        url '/path/to/DexGuard/lib'
    }
  }
  dependencies {
    classpath 'com.guardsquare:dexguard-gradle-plugin:+'
  }
}
    Avoid modifying your DexGuard installation

    In case you need to copy the DexGuard plugin to a different location, make sure to copy the whole directory structure located in /path/to/DexGuard/lib.

    Configuring the DexGuard Gradle plugin¶

    Now that the DexGuard plugin is added to your project, you can apply and configure DexGuard using a dexguard configuration block. This is a top-level configuration block and should be placed outside the android block.

    The snippets below show the minimum configuration required to execute DexGuard automatically for the release variant of an application.

    app/build.gradle.kts
    plugins { 
    alias(libs.plugins.dexguard)
}

dexguard {
    path = "/path/to/DexGuard"
    license = "/path/to/dexguard-license.txt"
    configurations {
        register("release") {
            defaultConfiguration("dexguard-release.pro")
        }                               
    }
}
    gradle/libs.versions.toml
    [versions]
dexguard = "+"

[plugins]
dexguard = { id = "com.guardsquare.dexguard", version.ref = "dexguard" }

    app/build.gradle
    plugins {
    alias(libs.plugins.dexguard)
}

dexguard {
  path = '/path/to/DexGuard'
  license = '/path/to/dexguard-license.txt'
  configurations {
    release {
      defaultConfiguration 'dexguard-release.pro'
    }
  }
}
    gradle/libs.versions.toml
    [versions]
dexguard = "+"

[plugins]
dexguard = { id = "com.guardsquare.dexguard", version.ref = "dexguard" }

    app/build.gradle
    apply plugin: 'com.guardsquare.dexguard'

dexguard {
  path = '/path/to/DexGuard'
  license = '/path/to/dexguard-license.txt'
  configurations {
    release {
      defaultConfiguration 'dexguard-release.pro'
    }
  }
}

    The path option specifies where to find your DexGuard installation. If you are using the maven distribution, this option is not required.

    The license option indicates where your license file can be found.

    In the configurations block, you should specify which build variants you want DexGuard to protect. In the example snippets, the release variant is configured.

    The default configuration defaultConfiguration contains pre-defined, required configuration options.

    Disable ProGuard & R8!

    DexGuard expects its inputs to be unobfuscated! Make sure the variants you want to protect have the minifyEnabled and shrinkResources options set to false in the android Gradle block. 

    android {
    buildTypes {
        release {
            minifyEnabled false
            shrinkResources false
        }
    }
}

    At this point, DexGuard is completely integrated in your build process and will be executed when you build your application. Please verify your setup by building your application (./gradlew bundle for a bundle or ./gradlew assemble for an apk).

    If everything was set up correctly, a protected version of your application was created alongside the unprotected version with a -protected suffix.

    Unable to build your application?

    To verify that your environment is set up correctly, try and build one of the samples included in the DexGuard distribution. Compare the setup of the samples to your own.

    The build/outputs/dexguard directory contains several other outputs. Most notably, a Protection Report can be found under build/outputs/dexguard/report. This report indicates the level of protection of your application and will be your main guideline when configuring additional protections later.

    The Gradle plugin will automatically configure DexGuard to sign the protected app using the same key that was used to sign the original app. So you don't have to explicitly configure DexGuard to sign the processed app.

    Default Android Studio install task

    The Android Gradle Plugin install task will install the unprotected application. If you want to test the protected application instead, you can use adb to install the protected application from the command line. E.g. adb install build/outputs/apk/release/MyApp-release-protected.apk.

    Alternatively, consider using the ** overwriteUnprotected ** option to overwrite the unprotected version. The protected apk will then no longer have the -protected suffix, and will be recognized by the install task. This may cause unexpected Gradle caching results!

    Automated configuration phase¶

    At this point, you have completed your first successful protected build, congratulations! Go ahead and try running it on an emulator or test device.

    Did your protected application crash? Do not worry, this is expected!

    You are likely facing runtime crashes due to reflection in your application. Reflection uses strings to find entities directly, but since DexGuard has obfuscated the names of these entities, the reflection calls in your application are not able to find the referred entities, and are throwing various forms of exceptions that indicate some entity could not be found.

    To solve this issue, all entities that are accessed by reflection should be targeted by keep rules. These keep rules preserve reflective accesses by keeping the names of the entities.

    Reusing your existing ProGuard & R8 configurations

    DexGuard is backward compatible with ProGuard and R8. As such you can reuse your ProGuard configuration files. However, inclusion of a proguard-project.txt configuration is not mandatory.

    Manually figuring out the required set of keep rules can only be done one keep rule at a time:

    1. Build, launch and explore the application until it crashes,
    2. Check which entity was reflected and caused the crash,
    3. Add a keep rule for that entity,
    4. Rebuild the application and repeat the process.

    This is a very time-consuming effort. Instead, DexGuard can build an instrumented version of your application that automatically collects the required keep rules and does not crash at runtime. This feature enables you to collect all keep rules using a single build, rather than having to repeat the process for each keep rule!

    Additionally, this instrumented build gathers performance data which DexGuard leverages during the protection run to ensure acceptable performance impact when injecting Runtime Application Self Protections (referred to as RASP). This will become relevant when setting up advanced protections later.

    To build the instrumented version, simply follow the next steps:

    1. Add a dexguard-project.txt file to your project.
    2. Add the -addconfigurationdebugging and -addprofiling configuration options to the configuration.
      dexguard-project.txt
      -addconfigurationdebugging
-addprofiling

    3. Include the configuration when running DexGuard.

      DexGuard plugin block
      dexguard {
  path = '/path/to/DexGuard'
  license = '/path/to/dexguard-license.txt'
  configurations {
    release {
      defaultConfiguration 'dexguard-release.pro'
      configuration '/path/to/dexguard-project.txt'
    }
  }
}

      bin/dexguard.sh -injars path/to/my-application.apk \
                -outjars path/to/obfuscated-application.apk \
                -androidsdk path/to/android/sdk \
                -include /lib/dexguard-release.pro \ 
                -include path/to/dexguard-project.txt \
                -keystore path/to/keystore \
                -keyalias your_key_alias \
                -keypassword your_key_password \
                -keystorepassword your_keystore_password

    Why your instrumented application looks different

    • This configuration only performs name obfuscation. Advanced protection features are disabled.
    • Your application is prevented from being released on the Google Play Store.
    • The icon of your application will be changed to the DexGuard icon to indicate an instrumented build.
    • Once you have your debug configuration, you should discard this build and remove the options from your configuration.

    Once your instrumented application is built, run it on a device and extensively explore your application. In the logcat, your application will now print keep rule recommendations whenever a reflection call is encountered. Additionally, these keep rules and some performance metrics are collected to files on the device. Instructions for retrieving these files can be found in logcat and will have the following formats:

    Writing rule suggestions to <path>: use `adb pull <path>` to retrieve the file from the device.
Writing profiling data to device storage. Use `adb pull <path>` to retrieve the file.

    After running the suggested adb pull commands in your project directory, you will have:

    • a dexguard-debugging-rules.txt file to prevent runtime crashes due to reflection.
    • a profilingdata.pb file to manage performance impact when injecting RASP.

    Explore extensively!

    It is important that you explore your instrumented application thoroughly. Unexplored code paths will not be covered by the generated configuration and may remain unstable!

    To conclude the automated configuration phase:

    1. Remove the instrumentation options from your dexguard-project.txt file.

    2. Include the generated dexguard-debugging-rules.txt configuration.

      DexGuard plugin block
      dexguard {
  path = '/path/to/DexGuard'
  license = '/path/to/dexguard-license.txt'
  configurations {
    release {
      defaultConfiguration 'dexguard-release.pro'
      configuration '/path/to/dexguard-debugging-rules.txt'
      configuration '/path/to/dexguard-project.txt'
    }
  }
}

      bin/dexguard.sh -injars path/to/my-application.apk \
                -outjars path/to/obfuscated-application.apk \
                -androidsdk path/to/android/sdk \
                -include /lib/dexguard-release.pro \ 
                -include path/to/dexguard-debugging-rules.txt \
                -include path/to/dexguard-project.txt \
                -keystore path/to/keystore \
                -keyalias your_key_alias \
                -keypassword your_key_password \
                -keystorepassword your_keystore_password

    That concludes the basic configuration phase! You should now be able to build a minimally protected version of your application that doesn't crash at runtime. If your application still crashes, perhaps the troubleshooting page can offer a solution.

    Improve your protection¶

    Now that you have your first stable protected build, it is time to improve the security of your application.

    An invaluable resource to do that is the Protection Report we mentioned earlier. If you have integrated DexGuard in your Gradle process, the report is located at build/outputs/dexguard/report. Users of standalone mode first have to specify the location where the report can be generated.

    bin/dexguard.sh -injars path/to/my-application.apk \
                -outjars path/to/obfuscated-application.apk \
                -androidsdk path/to/android/sdk \
                -include /lib/dexguard-release.pro \ 
                -include path/to/dexguard-debugging-rules.txt \
                -include path/to/dexguard-project.txt \
                -protectionreportdirectory /path/to/report-directory \
                -keystore path/to/keystore \
                -keyalias your_key_alias \
                -keypassword your_key_password \
                -keystorepassword your_keystore_password

    Depending on the threat model for your application, prioritize your work on different protection techniques and improve your score in relevant risk categories.

    We recommend the following iterative approach when enabling advanced protection features.

    1. Identify your most relevant security metric(s) in the Protection Report, and discover which of DexGuard's features contribute to it.
    2. Click on a disabled feature to get redirected to the relevant section in the manual.
    3. Configure the feature in your dexguard-project.txt file by following the manual.
    4. Rebuild your application.
    5. Test the application for stability and performance.
    6. Tune the configuration to fix any issues prior to enabling the next protection feature.
    7. Repeat the previous step for the next feature.

    Layered protection

    The different protection features reinforce each other. This is why setting up all the protection layers is important. We recommend scoring grade A in every category in the Protection Report.

    Monitor your application¶

    We recommend the iterative approach because it gives you an indication of the impact a particular feature has on your application. This way, when things go wrong, you know exactly what feature caused the issue, and you can reconfigure it accordingly.

    Below, we'll touch on the most common issues you may encounter when configuring advanced protections.

    Poor app performance¶

    The most common issue you may encounter at this stage is app performance problems. This is expected and can often be easily addressed.

    The most common reason for decreased app performance is that you have enabled a feature with a high performance impact on a performance-sensitive class or method.

    Most notably, while our RASP checks are very powerful, some are very expensive (e.g. the hook detector). This is where the profilingdata.pb file, generated by the instrumentation build, will help. Add the -profilingdata /path/to/profilingdata.pb option to your configuration as an initial indication of performance-sensitive code for DexGuard. If you notice performance is still poor, you can exclude specific performance-sensitive areas of your code from being injected into by using the -raspdenylist option.

    You can find an impact table for each feature on the protection overview page. Limit the use of features with medium to high performance impact to the security-sensitive parts of your code.

    Application size increase¶

    You may find that the size of your application has increased too much. Double check that you haven't been too aggressive with the applied level of hardening of the feature you are currently tuning.

    • All encryption features require DexGuard to inject decryption code into your application, so they will have a direct impact on your app's size. You might consider narrowing down the allowlist for which you enabled such features. Encrypting all strings or classes is overkill; focus on the most security-sensitive ones instead.
    • DexGuard's RASP features inject code snippets with small checks into your application. You can reduce the amount of injected checks by reducing the aggressiveness level.
    • Code obfuscation has an option to tweak the aggressiveness level. Find a balance in setting it higher for security-sensitive methods, lower for other methods.

    Testing¶

    It is important to test your application but there are certain limitations to be aware of when testing applications protected by DexGuard.

    Unit testing is not possible because

    • DexGuard changes the internals of your application.
    • DexGuards features rely on the Android environment and this cannot easily be mocked.

    Integration testing is only possible with end-to-end tests because checking internal application state is not possible.

    Supported integration testing frameworks
    • Espresso
    • Appium
      • UI elements can be selected using the UI tree.
      • Element IDs can be used only if kept by adding additional keep rules to your configuration.

    Before you go live¶

    Once your protection configuration is complete and everything is working properly, it’s time to plan your go-live. Consider the following best practices in your go-live strategy:

    Release only well-protected versions¶

    If you have implemented protection only partially, we strongly recommend you not to release such a version into production.

    The reason for this is that partially protected versions will teach reverse engineers the methods that you are using by introducing protection gradually, when it can be digested more easily.

    Having all protections implemented at once significantly raises the bar for the attackers.

    Regularly adopt new versions of DexGuard!

    Due to DexGuard's polymorphic approach, every new release of your app will be hardened differently. As such, sunsetting older versions of your app at a regular cadence is always a good practice, as it will force adversaries to restart their efforts again from scratch. Additionally, our RASP checks are under constant development. To stay up to speed with the attack surface, it is important that you update DexGuard and rebuild your protected application regularly.

    Check your crash reporting¶

    Your application is now protected, and one of the most important protection techniques used in DexGuard is name obfuscation. This means that the names of classes and class members will be changed, which makes it difficult for reverse engineers, but it can also make it difficult for you to diagnose crashes.

    Guardsquare tools enable you to de-obfuscate crash stack traces automatically or semi-automatically. However, you need to verify that everything is configured correctly before you go live by simulating a crash and making sure you are able to get the de-obfuscated trace.

    Consider threat reporting¶

    If you have configured RASP features, Guardsquare will detect various tools and techniques used to tamper with your application or use it in a compromised environment. By default, this information stays local within the app, however you may want to benefit from centralizing this information and using it in your fraud monitoring process.

    Enabling Guardsquare's mobile threat monitoring solution ThreatCast is very straightforward and will bring a lot of visibility and useful insights into the attack vectors your application is suffering from after it has been released into the wild.

    Plan deprecation of your unprotected application¶

    If you have an unprotected version of your application in production, it can still be attacked. Plan an approach to phase it out so that only the properly protected version of the app is out there.

    We will always be happy to help you on all stages of your journey - from early onboarding to post-going-live. Do not hesitate to reach out to us if you have any questions or suggestions for our team.
